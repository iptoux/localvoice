# Internationalization (i18n) — LocalVoice

This document describes the i18n approach for LocalVoice, including how to add UI strings, where translation files live, and how language-specific settings work.

## Overview

LocalVoice uses **react-i18next** for internationalization in the frontend. All user-facing strings should be externalized to translation files rather than hardcoded in components.

**Current status:** Translation files and infrastructure are being prepared. Until fully implemented, UI strings follow a naming convention that makes future migration straightforward.

## Translation Files

Translation files live in `src/i18n/`:

```
src/i18n/
├── index.ts          # i18n configuration
├── en.json           # English strings
├── de.json           # German strings
└── types.ts          # Namespace type definitions
```

## Adding New UI Strings

### Step 1: Add the Key to Translation Files

For each new string, add entries to both `en.json` and `de.json`:

```json
// src/i18n/en.json
{
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    "delete": "Delete"
  },
  "dashboard": {
    "title": "Dashboard",
    "totalWords": "Total Words"
  }
}
```

```json
// src/i18n/de.json
{
  "common": {
    "save": "Speichern",
    "cancel": "Abbrechen",
    "delete": "Löschen"
  },
  "dashboard": {
    "title": "Übersicht",
    "totalWords": "Gesamtwörter"
  }
}
```

### Step 2: Use the Translation in Components

```tsx
import { useTranslation } from 'react-i18next';
import '../i18n';

function Dashboard() {
  const { t } = useTranslation();
  
  return (
    <div>
      <h1>{t('dashboard.title')}</h1>
      <p>{t('dashboard.totalWords')}: 1234</p>
    </div>
  );
}
```

### Namespace Conventions

Organize translation keys by feature area:

| Namespace | Contents |
|-----------|----------|
| `common` | Shared buttons, labels, actions |
| `pill` | Pill component states and labels |
| `sidebar` | Navigation items |
| `dashboard` | Dashboard page |
| `history` | History page |
| `dictionary` | Dictionary page |
| `models` | Model manager |
| `settings` | Settings page |
| `errors` | Error messages |
| `onboarding` | Onboarding flow |

## Language Types

### Transcription Language

Transcription language is separate from UI language. It determines which Whisper model/language to use for speech recognition.

**Type definition** (`src/types/index.ts`):

```ts
export type TranscriptionLanguage = 
  | 'auto'   // Auto-detect
  | 'de'     // German
  | 'en'     // English
  | 'fr'     // French
  | 'es'     // Spanish
  | 'it'     // Italian
  | 'pt'     // Portuguese
  | 'nl'     // Dutch
  | 'pl'     // Polish
  | 'ru'     // Russian
  | 'ja'     // Japanese
  | 'zh';    // Chinese
```

**Stored setting:** `transcription.default_language`

### UI Language

UI language determines which translation file to use for the interface.

**Supported languages:**

| Code | Language |
|------|----------|
| `en` | English |
| `de` | German |

**Stored setting:** `app.language` (planned)

## Language-Specific Transcription Settings

### Per-Language Model Defaults

Users can set a different default model for each transcription language:

| Setting Key | Example Value |
|-------------|---------------|
| `transcription.default_model_de` | `ggml-base-de.bin` |
| `transcription.default_model_en` | `ggml-base.bin` |

### Settings Page Integration

The settings page uses `ALL_LANGUAGES` array for transcription language selection:

```ts
// src/pages/Models.tsx
export const ALL_LANGUAGES = [
  { value: 'auto', label: 'Auto-detect' },
  { value: 'de', label: 'Deutsch' },
  { value: 'en', label: 'English' },
  // ... other languages
] as const;
```

### Dictionary Language Filter

Dictionary entries and correction rules can be filtered by language:

```ts
// src/pages/Dictionary.tsx
export const LANGUAGES = ['de', 'en'] as const;
```

## Future Multi-Language Support

The following changes will enable additional UI languages:

### 1. Add New Translation File

```bash
# Create new translation file
cp src/i18n/en.json src/i18n/fr.json
# Then translate all values to French
```

### 2. Register in i18n Configuration

```ts
// src/i18n/index.ts
import fr from './fr.json';

i18n
  .use(Backend)
  .init({
    resources: {
      en: { translation: en },
      de: { translation: de },
      fr: { translation: fr },  // Add new language
    },
  });
```

### 3. Update Language Selector

Add new language to the UI language selector in settings.

### 4. Update Types

```ts
// src/i18n/types.ts
export type UILanguage = 'en' | 'de' | 'fr';
```

## Migration Guide: Extracting Hardcoded Strings

When migrating existing components to use translations:

1. **Identify hardcoded strings** in components
2. **Create keys** in appropriate namespace
3. **Add translations** for EN and DE
4. **Replace strings** with `t('key')` calls
5. **Handle dynamic values** with interpolation:

```tsx
// Before
<p>Total: {count} words</p>

// After
<p>{t('dashboard.totalWords', { count })}</p>
```

```json
// en.json
{
  "totalWords": "Total: {{count}} words"
}
```

### Dynamic & Plural Handling

```tsx
// Single form
<p>{t('session.count', { count: 1 })}</p>

// Use 'one' and 'other' for plurals
{t('session.count', { count: sessions.length, defaultValue: "{{count}} sessions" })}
```

## Implementation Checklist

When implementing full i18n support:

- [ ] Install `react-i18next` and `i18next`
- [ ] Create `src/i18n/index.ts` with configuration
- [ ] Create `src/i18n/en.json` with all current strings
- [ ] Create `src/i18n/de.json` with German translations
- [ ] Add `app.language` setting to database
- [ ] Create language selector in settings
- [ ] Wrap app with `I18nextProvider`
- [ ] Extract strings from all pages/components
- [ ] Update Zustand stores for language preference
- [ ] Persist language preference in settings
